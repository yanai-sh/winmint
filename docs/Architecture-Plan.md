# WinMint Architecture Plan

Branch: `architecture/profile-engine`

## Purpose

This document defines the target architecture for WinMint. The goal is not a full rewrite. The goal is to turn the current working project into a professional, modular, profile-driven Windows ISO builder without letting the repo fragment into hundreds of tiny scripts.

WinWS should remain:

- Script-first and transparent.
- Launchable through `irm https://winmint.yanai.sh | iex`.
- Windows-native in implementation.
- Opinionated in product decisions.
- Auditable in every build result.

## Core Architecture

The main design rule:

```text
UI creates intent.
Engine performs work.
Reports explain work.
FirstLogon finishes live-user setup.
```

Target flow:

```text
Bootstrap
  -> launches UI or CLI

UI / CLI
  -> creates BuildProfile.json
  -> validates BuildProfile.json

Engine
  -> consumes BuildProfile.json
  -> builds ISO
  -> writes BuildManifest.json
  -> writes logs

Windows Setup
  -> consumes staged autounattend/setup scripts
  -> runs SetupComplete
  -> reaches official OOBE

FirstLogon Agent
  -> consumes BuildProfile.json subset
  -> runs live-user setup
  -> writes AgentState.json
  -> writes logs
```

No UI event handler should directly decide low-level build behavior. UI code may collect input, preview outcomes, and call profile services. The profile resolver owns derived defaults.

## Separation of Concerns

| Area | Owns | Must Not Own |
|------|------|--------------|
| Bootstrap | Fetching/releasing the app bundle, basic trust checks, launch path | Build decisions, UI state, debloat logic |
| UI | Guided input, previews, validation messages, profile creation | DISM calls, WIM servicing, registry hive edits |
| Profile | Defaults, derived settings, schema validation, compatibility checks | Mounting images, installing packages |
| Engine | ISO extraction, WIM servicing, drivers, staged setup files, output ISO | WPF controls, user interaction, live-user app installs |
| Setup scripts | Machine-level setup phases during Windows install | User preference prompts, package source policy |
| FirstLogon Agent | Live-user setup, WSL, editors, shell layers, retry state | Offline image servicing, destructive disk choices |
| Reporting | Manifest, logs, user-readable summaries | Business logic decisions |
| Assets | Versioned payloads, configs, icons, cursors, Windhawk data | Executing setup logic |

## Contracts

The project should converge on three first-class JSON contracts.

### BuildProfile.json

Generated by the UI or CLI before the build starts. This is the single source of truth for user intent.

Owns:

- Source ISO path and detected architecture.
- Target device mode: `ThisPC` or `DifferentPC`.
- Edition mode: `TargetLicense` or `Fixed`.
- Disk partitioning mode.
- Driver source mode.
- Local account/device identity.
- Desktop shell layers.
- WSL distro selection.
- IDE/editor selections.
- Feature toggles that are truly user choices.

Does not own:

- Build timestamps.
- WIM mount paths.
- Downloaded file hashes.
- Step success/failure state.
- UI-only display strings.

### BuildManifest.json

Generated by the engine after or during a build. This explains what actually happened.

Owns:

- Source ISO metadata.
- Serviced image indexes.
- Edition mode and image names.
- Removed packages/capabilities.
- Injected drivers.
- Staged setup scripts.
- Staged payloads with versions and hashes.
- Warnings and risk flags.
- Output ISO path.
- Build duration and tool versions.

### AgentState.json

Generated by the FirstLogon Agent on the installed system. This makes first-run setup resumable.

Owns:

- Step status: `pending`, `running`, `done`, `failed`, `skipped`, `retryable`, `needsReboot`.
- Attempt counts.
- Last error per step.
- Installed package IDs and versions when available.
- Reboot recommendation.
- Completion marker.

### `src/WinWS`

PowerShell engine module. This is where Windows servicing lives.

Target internal layout:

```text
src/WinWS/
|-- WinWS.ps1
|-- Public/
|   |-- New-WinWSBuildProfile.ps1
|   |-- Test-WinWSBuildProfile.ps1
|   |-- Start-WinWSBuild.ps1
|   `-- New-WinWSBuildReport.ps1
`-- Private/
    |-- Config/
    |-- Drivers/
    |-- Image/
    |-- Media/
    |-- Payloads/
    |-- Reporting/
    |-- Setup/
    `-- Validation/
```

Rules:

- Public functions are stable entrypoints.
- Private functions are grouped by domain.
- Keep orchestration separate from implementation.
- `Pipeline.ps1` should become a thin orchestrator, not a giant logic file.
- Image servicing code should not know WPF exists.

### `src/WinWS.UI`

PowerShell-hosted WPF app. This collects choices and renders previews.

Target internal layout:

```text
src/WinWS.UI/
|-- Start-WinWSUI.ps1
|-- Views/
|-- ViewModels/
|-- Services/
|-- Resources/
`-- Profiles/
```

Rules:

- Views contain XAML only.
- ViewModels contain page state and simple commands.
- Services call profile APIs and engine APIs.
- UI logic should not directly mutate engine settings.
- Keep UI orchestration in `ViewModels/` + `Services/` (single cinematic stack); avoid parallel wizard implementations.

### `src/WinWS.Agent`

FirstLogon/live-user setup agent.

Target internal layout:

```text
src/WinWS.Agent/
|-- Start-WinWSAgent.ps1
|-- Agent.UI/
|-- Modules/
|   |-- Core/
|   |-- Desktop/
|   |-- Editors/
|   |-- Git/
|   |-- Packages/
|   |-- Shell/
|   `-- Wsl/
`-- State/
```

Rules:

- The worker is idempotent.
- The UI is a progress viewer/controller.
- Every step writes state before and after it runs.
- A failed optional layer must not break the entire first logon.
- Reboot-required steps must be explicit.

### `scripts`

Only developer and setup entry scripts belong here.

Allowed:

- Validation scripts.
- Release bundle scripts.
- One-off tooling under `scripts/tools/`.
- Windows setup phase entrypoints that must live as loose scripts.

Not allowed:

- Core engine logic.
- UI logic.
- Agent modules.
- Random helper scripts that should be module functions.

## File and Module Discipline

Avoid both extremes:

- One 2000-line script that nobody can reason about.
- Hundreds of 20-line files that create navigation noise.

Preferred file size:

| File Type | Target Size | Notes |
|-----------|-------------|-------|
| Orchestrators | 100-250 lines | High-level flow only. |
| Domain modules | 150-500 lines | Group closely related behavior. |
| XAML views | 150-700 lines | Split only when a view has real subviews. |
| Config JSON | Any reasonable size | Prefer data over code where possible. |
| Tests | 50-300 lines | Group by behavior, not by function. |

Rules:

- One file should have one clear reason to change.
- Do not create one file per tiny helper.
- Do not mix UI, profile resolution, and engine behavior in the same file.
- Prefer data catalogs for package/removal lists over hard-coded arrays in logic.
- Keep naming boring and predictable.

## Profile Resolution

The profile layer should normalize raw UI choices into concrete build behavior.

Example:

```text
Raw UI:
  TargetDevice = ThisPC
  DriverMode = Auto
  IsoArchitecture = arm64
  HostArchitecture = amd64

Resolved profile:
  TargetDevice = DifferentPC
  DriverMode = None
  RiskFlags += HostIsoArchitectureMismatch
```

The UI may preview this behavior, but profile resolution should be reusable by UI, CLI, tests, and future automation.

Suggested functions:

```powershell
New-WinWSBuildProfile
Resolve-WinWSBuildProfile
Test-WinWSBuildProfile
New-WinWSBuildConfig -Profile $profile
```

## Build Engine Phases

Engine orchestration should be explicit and phase-based.

Target phases:

1. Validate profile and environment.
2. Prepare workspace.
3. Mount or extract source ISO.
4. Discover image metadata.
5. Resolve target images.
6. Stage assets and setup scripts.
7. Service boot images.
8. Service install images.
9. Apply unattended setup files.
10. Write manifest and reports.
11. Create output ISO.
12. Cleanup mounts and workspace.

Each phase should:

- Log start/end.
- Emit structured result data.
- Fail with clear error categories.
- Cleanup its own temporary resources where possible.

Console logging direction:

- Use PwshSpectreConsole as the terminal UI layer instead of hand-built console formatting.
- Use `Write-SpectreHost` for rich line output.
- Use `Invoke-SpectreCommandWithStatus` for indeterminate setup phases.
- Use `Invoke-SpectreCommandWithProgress` for phases with measurable work.
- Use `Invoke-SpectreLive` only for true live dashboards; avoid making normal logs harder to copy.
- Keep engine progress as structured events so WPF, CLI, and FirstLogon can render the same events differently.

## FirstLogon Design

FirstLogon should be visible, but not chatty. It should feel like a short finishing stage after official Windows OOBE.

Recommended behavior:

- Launch once at first interactive logon.
- Show a compact WPF progress window.
- Run selected live-user steps in a worker.
- Allow viewing logs.
- Show partial success clearly.
- Offer retry for failed retryable layers.
- Recommend reboot when needed.

Do not use FirstLogon for:

- Offline WIM servicing.
- Destructive disk behavior.
- Edition selection.
- Low-level driver injection.

Use FirstLogon for:

- WSL distro setup.
- Editor installs.
- Windhawk/YASB/Komorebi live-user configuration.
- Git/dotfiles if explicitly configured.
- Final desktop/user-session polish.

## OOBE Policy

WinWS uses official Windows OOBE where it is the right owner, and preconfigures the rest through supported setup/profile contracts.

- The network page stays visible so the user can join Wi-Fi before first-logon automation.
- `AccountMode Local` creates the local administrator through `autounattend.xml`; the account OOBE page is not needed.
- `AccountMode MicrosoftOobe` removes local-account and account-bypass nodes so official Windows account setup is shown.
- Privacy is WinWS-owned, not a choice matrix. Location services are enabled by default and can be disabled with the explicit location setting; advertising, tailored experiences, feedback/activity history, and similar surfaces are disabled by baseline policy.
- Phone Link is not a profile group. Preserve/remove behavior is a feature policy decision, separate from whether Microsoft shows cloud-service prompts during official OOBE.

## Testing Strategy

Minimum test matrix:

| Path | Purpose |
|------|---------|
| x64 VM, no drivers | Fast baseline regression. |
| x64 VM, This PC drivers | Same-machine convenience path. |
| ARM64 target, Different PC | Cross-architecture safety path. |
| INF folder with drivers | Manual target-driver path. |
| No internet at first logon | Agent resilience and clear failure state. |
| Internet at first logon | Full developer workstation setup. |
| Multiple WSL distros selected | Developer layer integration. |
| Standard desktop only | Clean baseline install. |
| Windhawk/YASB/Komorebi selected | Shell layer composition. |

Automated tests should start with profile validation and pure functions. Full ISO builds can remain manual or scheduled until the engine is more stable.

## Migration Plan

Do this incrementally. Keep the app runnable after every step.

### Phase 1: Contracts

- Add JSON schemas for build profile, manifest, and agent state.
- Add profile creation/resolution functions.
- Make current UI settings export a `BuildProfile.json`.
- Keep existing engine settings adapter for compatibility.

### Phase 2: Engine Boundary

- Make the engine accept a resolved profile.
- Move scattered defaults into profile resolution.
- Thin out pipeline orchestration.
- Make reports use manifest data instead of recomputing summaries.

### Phase 3: UI Boundary

- Replace direct build-setting mutation with profile services.
- Introduce page view-model files only where they reduce coupling.
- Keep WPF and the current visual direction, but simplify handler ownership.

### Phase 4: FirstLogon State

- Add `AgentState.json`.
- Make each agent module idempotent.
- Add visible WPF first-run progress UI.
- Add retry/reboot/failure display.

### Phase 5: Package and Payload Trust

- Add payload manifest with source URLs, versions, and hashes.
- Make release bundles include the profile schemas and payload manifest.
- Make the build report list staged payloads and hashes.

### Phase 6: Cleanup

- Remove stale docs and obsolete scripts.
- Move one-off tools under `scripts/tools/` or out of version control.
- Enforce file-size and boundary checks in validation.

## Open Decisions

| Decision | Current Lean |
|----------|--------------|
| Should the UI save build profiles by default? | Yes, under `%APPDATA%\WinWS\profiles`. |
| Should first-logon be blocking? | Visible and compact, but not a full-screen blocker. |
| Should the CLI be first-class? | Yes, after the profile schema is stable. |
| Should payloads be bundled or downloaded? | Core small assets bundled; larger tools versioned and downloaded with hashes. |
| Should OEM driver downloaders exist? | Later. Keep stable manual and This PC paths first. |
| Should WinWS ever become a compiled app? | Only if script-first distribution becomes a real limitation. |

## Success Criteria

The architecture pass is successful when:

- The UI can save a complete build profile before starting.
- The engine can build from a profile without WPF loaded.
- The manifest explains the build without scraping logs.
- FirstLogon can resume after interruption.
- Adding a new shell layer or developer tool requires changes in one domain, not across the UI, engine, setup scripts, and docs.
- The repo has fewer ambiguous scripts, not more.
- A new contributor can identify where a change belongs within five minutes.
